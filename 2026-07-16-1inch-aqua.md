# 1inch Aqua ‚Äî Security Review of the Shared-Liquidity Core

**Target:** 1inch Aqua ("Shared Liquidity Layer"), `github.com/1inch/aqua`
**Scope reviewed:** `src/Aqua.sol`, `src/libs/Balance.sol`, `src/AquaApp.sol`, `src/AquaRouter.sol`
**Date:** 2026-07-16 ¬∑ **Verdict:** No exploitable bug found in the reviewed core.

---

## 1. What Aqua is

Aqua is a **non-custodial shared-liquidity layer**. Instead of makers depositing tokens into a pool, they keep funds in their own wallets, grant the Aqua contract an ERC-20 allowance, and *authorise* specific "apps" to pull against a committed balance. Apps (DEX routers, strategies) pull the maker's tokens directly `maker ‚Üí taker` at execution time.

The security-critical property is therefore not "are pooled funds safe" ‚Äî there are no pooled funds ‚Äî but "**can an app (or an arbitrary caller) pull more of a maker's tokens, or a maker they weren't authorised for.**" Everything below is aimed at that question.

## 2. The core ledger (`Aqua.sol`)

Four external functions manage a nested balance mapping
`_balances[maker][app][strategyHash][token] ‚Üí Balance`:

- **`ship(app, strategy, tokens, amounts)`** ‚Äî the *maker* (msg.sender) commits: for each token it sets the balance to `amounts[i]` and records `tokensCount`. Guarded by `require(balance.tokensCount == 0)` ‚Äî **strategies are immutable**; you cannot silently re-ship over an active one. No tokens move here.
- **`pull(maker, strategyHash, token, amount, to)`** ‚Äî the *app* (msg.sender) pulls:
  ```solidity
  Balance storage balance = _balances[maker][msg.sender][strategyHash][token];
  (uint248 prevBalance, uint8 tokensCount) = balance.load();
  balance.store(prevBalance - amount.toUint248(), tokensCount);   // (A)
  IERC20(token).safeTransferFrom(maker, to, amount);              // (B)
  ```
- **`push(maker, app, strategyHash, token, amount)`** ‚Äî anyone tops up a maker's committed balance, paying in: `safeTransferFrom(msg.sender, maker, amount)` and `balance += amount`. Requires `tokensCount > 0 && != _DOCKED`.
- **`dock(app, strategyHash, tokens)`** ‚Äî the maker revokes: sets balance to `0` with `tokensCount = _DOCKED (0xff)`.

## 3. The finding the scanners raise ‚Äî and why it's a false positive

Slither's `arbitrary-send-erc20` and Semgrep's `bad-transferfrom-access-control` both flag line **(B)**: `safeTransferFrom(maker, to, amount)` moves tokens *from an address that is a function parameter* (`maker`), with `to` also attacker-supplied. That is the exact signature of the classic "arbitrary-from drain." Here it is **gated**, on three independent axes:

1. **Caller binding.** The balance read at (A) is keyed on `msg.sender` as the app. An attacker can only pull `maker`'s tokens if `_balances[maker][attacker][‚Ä¶]` is non-zero ‚Äî i.e. the maker must have `ship`-ed *to the attacker's own address*. A maker shipping to a legit app never populates the attacker's slot.
2. **Amount bound.** `prevBalance - amount` at (A) is `uint248` subtraction under Solidity 0.8.30 ‚Äî it **reverts on underflow**. You cannot pull more than the committed balance.
3. **Allowance + wallet cap.** (B) is a real `transferFrom` against the maker's ERC-20 allowance and wallet balance. Even a mis-committed balance can't move tokens the maker hasn't approved or doesn't hold.

So the "`to` is arbitrary" observation is by design (the app directs the maker's committed liquidity to a pool), and it is bounded by an authorisation the maker sets on-chain via `ship`. **False positive.**

### The `push` inflation question
The shipped `amounts` look like they should be a hard cap, but `push` lets *anyone* raise a balance. Does that let a malicious app over-pull? I worked the ledger both directions: `push` transfers `amount` **to the maker** while crediting the app the same `amount`. To inflate an app's pullable balance by X you must first give the maker X tokens; the app can then pull that X back ‚Äî net zero on the pre-existing funds. The maker's own balance beyond the shipped intent is never reachable, because the maker's *wallet balance* is the ultimate ceiling on any `transferFrom`. No over-pull.

## 4. Packed storage (`Balance.sol`)

`Balance` packs `uint248 amount + uint8 tokensCount` into one slot via assembly:
```solidity
function store(...) { let packed := or(amount, shl(248, tokensCount)); sstore(slot, packed); }
function load(...)  { amount := and(packed, 0x00ff..ff); tokensCount := shr(248, packed); }
```
The mask/shift split is clean, and every caller feeds a value that has already passed `SafeCast.toUint248()` or checked `uint248` arithmetic, so `amount`'s top 8 bits are always zero ‚Äî no collision with `tokensCount`. No bit-packing bug.

## 5. App-layer reentrancy (`AquaApp.sol`)

`AquaApp` is the base for apps built on Aqua. Its swap-verification primitive, `_safeCheckAquaPush`, confirms the taker pushed enough back to the maker after a pull. Critically it **requires the transient-storage reentrancy lock to be held**:
```solidity
require(_reentrancyLocks[maker][strategyHash].isLocked(), MissingNonReentrantModifier());
```
This blocks the nested-swap manipulation you'd otherwise use to fool a snapshot balance check, and the `isLocked()` guard catches apps that forget the `nonReentrantStrategy` modifier ‚Äî a nice fail-closed default. The balance check itself is snapshot-relative, so a front-run `push` can't substitute for the taker's payment.

## 6. Deployment composition (`AquaRouter.sol`)

`AquaRouter is Aqua, Simulator, Multicall, Rescuable` ‚Äî the core plus three 1inch standard mixins. The **non-custodial** design is what makes all three safe:

| Mixin | Usual risk | Why safe here |
|---|---|---|
| `Multicall` | msg.sender / msg.value abuse | `delegatecall(address(this))` **preserves msg.sender** (no privilege escalation); Aqua is non-payable ERC-20 (no `msg.value` double-spend) |
| `Simulator` | arbitrary delegatecall | always **reverts** with the simulation result ‚Üí any state change rolls back |
| `Rescuable` | owner drains user funds | **nothing to drain** ‚Äî the router never custodies user tokens; only stray/donated tokens are recoverable |

The `controlled-delegatecall` Slither flag resolves to `Simulator.simulate` inside `@1inch/solidity-utils` ‚Äî an off-chain `eth_call` gas-estimation helper, not attacker-reachable state. False positive.

## 7. Verdict

The reviewed core is a **well-architected non-custodial design**. Every scanner high-severity flag traces to either a 1inch standard dependency (`solidity-utils`: Simulator, UniERC20, Multicall) or the properly-gated `pull`. The obvious attack surfaces ‚Äî arbitrary drain, over-pull, reentrancy, token-scope escape, packed-storage corruption, unsafe mixin composition ‚Äî are each closed.

**No exploitable bug found in `Aqua` + `Balance` + `AquaApp` + `AquaRouter`.**

**Honest limits of this review:** single-pass manual read, no fork-test PoCs. Not covered: the individual "app"/strategy contracts built *on* Aqua (where custom logic ‚Äî and the delegatecall execution ‚Äî actually lives), and the full `@1inch/solidity-utils` dependency (standard, separately audited). A drain, if one exists, would have to live in the custody core reviewed here, and that layer is clean.
