# Moonwell — Triaging 16 High-Severity Flags on a Heavily-Audited Fork

**Target:** Moonwell, Code4rena ($100k max critical) — Compound-v2 + Aave-stkToken fork on Base / Optimism / Moonbeam
**Scope reviewed:** Static-analysis triage of all high-severity detector output + manual surface map
**Date:** 2026-07-16 · **Verdict:** All 16 high-severity flags = false-positive or admin-only/low.

---

## 1. Why this one is a triage exercise, not a bug hunt

Moonwell is a 3×+ audited fork of two of the most-forked codebases in DeFi (Compound v2 for the lending core, Aave's stkAAVE for the staking/safety module). On a target like this, **the static analyzer will always fire, and it will almost always be wrong** — because the "dangerous" patterns it detects (proxy delegatecall, arbitrary sends, uninitialized state) are the *intended* patterns of those well-known designs.

The skill here isn't finding a bug in the Slither output — there isn't one. It's being able to look at 16 high-severity flags and correctly tie each to the design pattern that makes it a false positive, *without* re-deriving it from scratch, so real attention goes to the places scanners can't see. Here's every flag, and why.

## 2. The 16 flags

**`controlled-delegatecall` ×2** — `TokenSaleDistributorProxy.fallback()` and `MErc20Delegator.delegateTo()`. These are **standard proxy delegation** (Compound's Delegator pattern + a token-sale proxy). The implementation is admin-set, not user-controlled — the exact false-positive twin of a real delegatecall-injection bug. **FP.**

**`arbitrary-send-erc20` ×2** — `StakedToken.claimRewards()` / `StakedTokenV2.claimRewards()` calling `safeTransferFrom(REWARDS_VAULT, to, amount)`. The `from` is flagged as arbitrary, but it's a **fixed `REWARDS_VAULT` constant** — this is Aave's stkAAVE reward-claim pattern verbatim, not an attacker-chooseable source. **FP** on the transfer. *(Note: the real surface here is the `amount` calculation, not the transfer — see §3.)*

**`arbitrary-send-eth` ×4** — `TemporalGovernor._executeProposal` and `MultichainGovernorV2._functionCallWithValue` are **governance executing arbitrary calls**, which is the entire point of a governor (intended, timelocked). `OEVProtocolFeeRedeemer._wrapNativeToWETH` just wraps native → WETH. `WethUnwrapper.send` is a stateless unwrap-and-forward helper. All **FP / low**.

**`unchecked-transfer` ×2** — `Comptroller._rescueFunds` (`onlyAdmin` rescue) and `EcosystemReserve.transfer` (controller-only). Admin-gated, privileged-by-design. **Low**, not a vulnerability.

**`uninitialized-state` ×5** — `MultichainGovernor.{xWell, well, distributor}` and `TokenSaleDistributorProxyStorage.admin` are set in an upgradeable `initialize()`, which Slither's dataflow **doesn't track** (it misses initializer writes on proxied contracts). `MultiRewardDistributor.marketConfigs` is a mapping — no initialization needed. **FP** (this is the single most common false positive on any upgradeable codebase).

**`reentrancy-eth` ×1** — not deep-verified, but on a Compound fork the guarded functions carry the inherited `nonReentrant` modifier; almost certainly **FP**. Flagged honestly as *probable* FP rather than confirmed.

**Result: the automated pass is clean.** No high-severity flag survives triage as a real bug. Exactly what you'd expect from a 3×-audited fork.

## 3. Where a real bug would actually live

This is the part that matters, and the part scanners are structurally blind to. If Moonwell has an exploitable bug, it is a **logic or economic** bug, in one of three places:

1. **Reward accounting.** `StakedToken.claimRewards`'s `amount` calculation and `MultiRewardDistributor`'s index/emission math (multiple reward tokens per market). This is the classic Compound-fork over-claim / index-manipulation surface — the transfer is fine, but is the *entitlement* computed correctly across market entry/exit, index updates, and multi-reward interleaving?
2. **Cross-chain governance.** `MultichainGovernor` / `TemporalGovernor` and the Wormhole VAA handling. The interesting questions are replay across chains, chain-id binding, and quorum/threshold edge cases at the message-verification boundary — none of which a single-contract static analyzer can reason about.
3. **Liquidation & interest-accrual rounding** in the `MToken` core. Compound-fork rounding directionality under specific utilisation/exchange-rate states.

None of these is reachable from Slither/Aderyn output. All three require reading the math against the invariants — and against a $100k critical bounty on a target this audited, the honest expected value is low. Which is exactly why the correct move after this triage is **not** to grind Moonwell deeper, but to point the same rigor at fresher, less-swept targets.

## 4. Verdict

**16 high-severity flags, all triaged to false-positive or admin-only/low.** The automated attack surface is clean. Documented so it isn't re-derived, and so that if Moonwell ships new code, only the *diff* — not the whole codebase — needs re-checking.

**Honest limits:** this is a static-analysis triage plus a manual surface map, not a line-by-line manual audit of the reward-accounting and cross-chain-governance math (§3), which is where genuine effort would go if the economics justified it. On a 3×-audited fork with a $100k ceiling, they don't — and saying so is part of the job.
